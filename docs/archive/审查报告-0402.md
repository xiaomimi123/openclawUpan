## 一、已修复的问题（共 14 项）

### 第一轮：代码逻辑问题（8 项）

#### HIGH-1: repair-config 用脱敏 API Key 重建配置 ✅

**文件**: `main.js` repair-config handler
**问题**: `save-setup` 将 API Key 存为 `***` 到 setup.json。当 `repair-config` 从 setup.json 重建 openclaw.json 时，`***` 被写入为真实 API Key。`refreshGatewayToken()` 成功后返回 `{ ok: true }`，用户看到"修复成功"，但 API 调用全部失败。
**修复**: 重建后立即返回 `{ ok: false, needsReconfig: true }`，提示用户重新输入 API Key。

#### HIGH-2: zip 解压 readStream 无 error handler ✅

**文件**: `main.js` installToLocal + install-weixin-plugin
**问题**: `zipfile.openReadStream` 的 readStream 没有 error handler。`pipe()` 不转发 read 端错误。如果 zip 条目读取出错，解压永久卡住，直到 5 分钟超时。
**修复**: 两处解压均添加 `readStream.on('error', ...)`。

#### HIGH-3: 修改 API Key 弹窗缺少 anthropic/openai ✅

**文件**: `launcher.html` modal-apikey
**问题**: 下拉菜单没有 anthropic 和 openai 选项。初始配置选了 anthropic 的用户，改 Key 时被回退到"自定义接入"，需要手填 Base URL 和模型 ID。
**修复**: 补齐 anthropic/openai 选项，可选填代理地址，model ID 行自动隐藏。

#### MEDIUM-4: openUrl cmd 回退有命令注入风险 ✅

**文件**: `main.js` openUrl
**问题**: `shell.openExternal` 失败后回退到 `execFile('cmd', ['/c', 'start', '""', url])`。URL 中的 `&` 会被 cmd 解释为命令分隔符。
**修复**: 改用 `rundll32 url.dll,FileProtocolHandler`。

#### MEDIUM-5: start-openclaw 无互斥锁 ✅

**文件**: `main.js` start-openclaw handler
**问题**: `if (openclawProc) return` 和 `spawn()` 之间有时间窗口，网络自动重试与手动点击可能同时触发，spawn 两个 gateway。
**修复**: 添加 `startingOpenclaw` flag，所有退出路径重置。

#### MEDIUM-6: pet-preload.js listener 泄漏 ✅

**文件**: `pet-preload.js`
**问题**: `onStatusUpdate` 未调用 `removeAllListeners`，多次调用会累积 listener。
**修复**: 添加 `removeAllListeners('pet-status')`。

#### LOW-7: setPreflight 参数遮蔽外部变量 ✅

**文件**: `launcher.html`
**问题**: `function setPreflight(id, state, detail)` 的 `state` 参数遮蔽了外层 `state` 对象。
**修复**: 重命名为 `status` / `pfStatus`。

#### LOW-8: Anthropic 验证器发真实 API 请求 ✅

**文件**: `main.js` API_VALIDATORS
**问题**: 验证 anthropic key 时发 POST /v1/messages（max_tokens=1），产生费用。其他服务商只调 GET /models。
**修复**: 改为 GET /v1/models，与其他服务商一致。

### 第二轮：Windows 用户场景问题（6 项）

#### CRITICAL-9: 无单实例锁，双击启动两个实例 ✅

**文件**: `main.js` app.whenReady
**场景**: 用户双击 exe 时启动较慢，以为没反应再点一次 → 两个实例同时运行 → 两个 gateway 争抢端口 18789 → 一个报端口占用，界面状态混乱。
**影响**: 几乎所有 Windows 用户都可能遇到。
**修复**: 添加 `app.requestSingleInstanceLock()`，第二实例自动退出并聚焦已有窗口。

#### CRITICAL-10: 覆盖本机已有 openclaw 配置 ✅

**文件**: `main.js` save-setup, `src/paths.js` configDir
**场景**: 用户电脑已安装 openclaw（通过 npm 或其他渠道），`~/.openclaw/openclaw.json` 已有其自定义配置（agents、channels、plugins 等）。启动器的 `save-setup` 直接覆盖整个文件，用户原有配置全部丢失。
**影响**: 本机已有 openclaw 的用户。
**修复**:
- 写入前先备份 `.bak`
- 合并模式：保留用户已有的 agents（非 defaults）、channels、plugins，只覆盖启动器管理的字段
- repair-config 损坏重建前也先备份

#### CRITICAL-11: killOpenclaw 在 Windows 上不可靠 ✅

**文件**: `main.js` killOpenclaw
**场景**: 拔出 U 盘 → `killOpenclaw()` → `proc.kill('SIGTERM')` 在 Windows 上只杀主进程不杀子进程树 → node.exe 子进程成为僵尸 → 用户反复插拔 U 盘，僵尸进程累积，占用内存和端口。
**影响**: 每次拔盘都可能残留进程。
**修复**: 改用 `taskkill /T`（杀进程树），先温和终止，2 秒后强制终止 (`/F`)。

#### HIGH-12: 杀毒软件锁文件导致安装随机失败 ✅

**文件**: `main.js` copyDirRecursive
**场景**: 安装过程中复制 Node.js runtime 到本机 → Windows Defender 实时扫描新文件 → 短暂锁定 → `copyFile` 报 EBUSY/EPERM → 安装失败，用户看到"可能是杀毒软件"提示但不知道怎么办。
**影响**: 约 10% 的首次安装。
**修复**: `copyFileRetry` 函数，EBUSY/EPERM/EACCES/UNKNOWN 自动重试 3 次，间隔递增（500ms/1s/1.5s）。

#### HIGH-13: 企业环境 PowerShell 受限 ✅

**文件**: `main.js` 磁盘检查 / 端口查找, `src/license.js` 序列号读取
**场景**: 企业 Windows 的 PowerShell ExecutionPolicy 为 Restricted，`Get-PSDrive`、`Get-NetTCPConnection`、`Get-CimInstance` 全部失败 → 磁盘检查跳过 → 端口释放失败 → 序列号读取失败。
**影响**: 企业环境用户。
**修复**:
| 功能 | 原方案 | 新方案（优先） | 回退 |
|------|--------|---------------|------|
| 磁盘空间 | PowerShell Get-PSDrive | wmic diskfree | PowerShell |
| 端口查找 | PowerShell Get-NetTCPConnection | netstat -ano + tasklist | PowerShell |
| U 盘序列号 | PowerShell Get-CimInstance | vol 命令 | PowerShell |

#### MEDIUM-14: npm 镜像源强制淘宝，海外用户超时 ✅

**文件**: `main.js` buildEnv
**场景**: 海外用户安装微信插件 → npm 被强制走淘宝镜像 → 跨洋访问超时 → 插件安装失败。
**影响**: 所有非中国大陆用户。
**修复**: `isChinaEnv()` 通过系统时区 + 语言检测，仅中国大陆环境使用淘宝源，其他地区使用 npm 官方源。
